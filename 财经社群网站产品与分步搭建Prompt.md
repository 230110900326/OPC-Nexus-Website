# 财经社群与行业内容聚合平台建设方案

> 品牌口号：**洞察 · 链接 · 增长**  
> 文档版本：V1.0  
> 建设目标：搭建一个集财经资讯、政策聚合、视频洞察、社区论坛、行业活动和内容推荐于一体的专业财经社群网站。

---

## 1. 项目定位

本项目参考 OPC Community、OPC 圈等财经社群网站的内容组织方式，并吸收小红书的信息流发现体验和 Bilibili 的视频卡片体验，建设一个面向财经、产业、投资和企业经营人群的内容与交流平台。

平台的核心价值对应品牌口号：

- **洞察**：聚合新闻、政策、研究文章和视频内容，提供结构化摘要与行业分类。
- **链接**：连接用户、作者、机构、UP 主、行业话题和线上线下活动。
- **增长**：通过高质量信息、专业讨论、活动和会员服务帮助用户获得认知与业务增长。

首期不建设证券交易、投资顾问或代客理财功能，重点做好内容、社区、活动、审核、推荐和运营后台。

---

## 2. 用户角色

| 角色 | 主要权限 |
|---|---|
| 游客 | 浏览公开内容、查看热榜、搜索、查看活动 |
| 注册用户 | 评论、发帖、点赞、收藏、关注、举报、报名活动 |
| 认证作者/机构 | 发布专栏内容，拥有认证主页 |
| 版主 | 管理指定论坛板块、处理帖子和评论 |
| 内容编辑 | 编辑和发布新闻、文章、专题、视频摘要 |
| 运营管理员 | 管理用户、活动、推荐位、采集来源和数据报表 |
| 超级管理员 | 角色权限、系统配置、操作日志和全站管理 |

## 3. 网站信息架构

### 3.1 顶部导航

- 首页
- 资讯
- 政策
- 视频
- 洞察/专栏
- 社区
- 活动
- 热榜
- 搜索
- 登录/注册或个人中心

### 3.2 首页

首页建议依次展示：

1. 顶部导航和品牌口号。
2. 焦点内容与重要政策。
3. 推荐、最新、关注三个信息流标签。
4. 洞察精选与深度文章。
5. 热门视频与关注 UP 主更新。
6. 社区热议。
7. 即将开始的活动。
8. 推荐作者、机构和内容来源。
9. 邮件或微信订阅入口。
10. 免责声明、隐私政策、用户协议和联系方式。

### 3.3 资讯频道

建议设置以下行业分类：

- 宏观与政策
- 资本市场
- 科技与产业
- 创业与投融资
- 财富管理
- 企业出海
- 公司与商业
- OPC 精选

资讯卡片至少显示：

- 标题
- AI 摘要
- 首图
- 来源名称
- 发布时间
- 行业标签
- 热度
- 原文链接
- AI 摘要标识

点击卡片可先进入本站摘要详情页，再通过“阅读原文”跳转来源网站。后台也应支持配置为直接跳转原文。

### 3.4 政策频道

政策内容除普通文章字段外，还应展示：

- 发文机关
- 文号
- 发布日期
- 生效日期
- 适用地域
- 涉及行业
- 政策状态
- 政策要点
- 可能影响
- 政策原文链接

### 3.5 视频频道

视频来源包括 Bilibili、YouTube 和抖音。平台先在后台维护经过筛选的垂类 UP 主或机构账号，定期发现其新视频并同步到网站。

视频卡片显示：

- 16:9 封面
- 视频标题
- 视频平台与 UP 主
- 发布时间与时长
- 3—5 条核心观点
- 基于字幕或语音转写生成的内容概要
- 行业标签
- 公开互动数据
- 原视频链接

“内容概要”不能直接复制视频简介。生成优先级为：

1. 使用平台合法提供的字幕。
2. 使用创作者授权提供的字幕。
3. 在得到授权的情况下对音频进行语音识别。
4. 无法获得字幕或音频授权时，只展示平台简介并明确标注“暂无内容概要”。

条件允许时，可增加带时间点的内容章节。

### 3.6 社区论坛

建议设置以下板块：

- 宏观与政策

- 财富管理
- 出海与全球市场
- 行业研究与专业问答
- 社群活动与公告

基础功能包括：

- 举报、审核、封禁和操作留痕
- 按最新、热门、精华排序

### 3.7 活动中心

- 线上直播、线下沙龙、峰会和闭门交流
- 活动详情、时间、地点、嘉宾和人数限制
- 报名表单与报名状态
- 报名成功通知
- 报名名单导出
- 二维码签到能力预留

### 3.8 用户中心

- 个人资料与头像
- 职业、公司、行业等选填信息

## 5. 资讯采集与内容加工方案

### 5.1 采集范围

- OPC 社区和获得授权的相关行业网站
- 国务院及相关部委网站
- 中国人民银行
- 国家金融监督管理总局
- 中国证监会
- 国家统计局
- 国家发展改革委
- 商务部、财政部、税务总局
- 地方政府和地方金融管理部门
- 获得授权或允许聚合的财经媒体、研究机构和行业协会
- 白名单内的 Bilibili、YouTube、抖音账号

### 5.2 技术原则

采集优先级如下：

1. 官方开放 API。
2. RSS、Atom、Sitemap 或官方订阅机制。
3. 允许抓取的静态网页。
4. 必须渲染时才使用浏览器自动化。

不得绕过登录、验证码、签名校验、付费墙或平台反爬机制。上线前应检查目标网站的 robots.txt、服务条款、转载授权和图片版权。

### 5.3 处理流程

1. 后台配置来源、栏目、抓取方式、抓取频率和可信等级。
2. 定时发现新文章或新视频。
3. 提取标题、正文、首图、作者、来源、发布时间和原始链接。
4. 清洗广告、导航和无关文本。
5. 使用规范化 URL、标题相似度和正文指纹去重。
6. 提取行业、关键词、公司、机构、人物和地区。
7. 生成摘要、核心观点或政策解读字段。
8. 执行敏感词、低质量和合规检查。
9. 高可信来源自动发布，其他来源进入人工审核。
10. 定期检查链接失效、原文更新和内容下线。

### 5.4 首图策略

1. 优先使用来源授权的封面。
2. 其次使用网页开放元数据中允许引用的图片。
3. 无授权、图片失效或无首图时，生成带来源、分类和标题的品牌占位图。
4. 不应默认把外站图片永久复制到本站对象存储。

### 5.5 内容保存原则

默认只公开展示标题、摘要、首图、来源和原文链接，避免未经许可完整转载。为了生成摘要而临时处理的正文应设置保存期限和访问权限；如有完整转载授权，再由后台单独开启全文展示。

## 6. 热度与推荐机制

### 6.1 内容热度

先在同平台、同分类、相近发布时间的内容中对互动指标进行标准化，再计算互动分：

```text
互动分 =
  0.25 × 点赞标准分
+ 0.35 × 评论标准分
+ 0.40 × 转发标准分
```

综合热度：

```text
综合热度 =
  0.35 × 互动分
+ 0.25 × 时间新鲜度
+ 0.20 × 行业关键词匹配度
+ 0.10 × 来源可信度
+ 0.10 × 内容质量或编辑评分
```

时间衰减参考：

```text
时间新鲜度 = exp(-发布时间距今小时数 / 半衰期)
```

不同内容建议使用不同半衰期：

| 内容类型 | 建议半衰期 |
|---|---:|
| 快讯 | 12—24 小时 |
| 普通新闻 | 48 小时 |
| 视频 | 3—7 天 |
| 政策 | 7—15 天 |
| 深度文章 | 15—30 天 |

### 6.2 个性化推荐

```text
最终推荐分 =
  70% × 综合热度
+ 20% × 用户兴趣匹配
+ 10% × 内容多样性
```

用户兴趣可来自：

- 主动选择的行业
- 浏览、收藏、点赞和评论
- 关注的作者、机构和 UP 主
- 搜索关键词
- 内容停留时间

新用户使用“编辑精选 + 全站热榜 + 用户选择行业”的冷启动策略。

### 6.3 防刷与质量控制

- 同一账号重复互动降权
- 新账号短时间密集互动降权
- 异常 IP、设备和请求频率过滤
- 删除违规互动后重新计算热度
- 评论加入有效评论比例
- 低可信来源设置推荐上限
- 管理后台保留人工加权、降权和置顶能力

## 7. 技术架构

### 7.1 推荐技术栈

- 前端：Next.js、TypeScript、Tailwind CSS
- 业务后端：NestJS、TypeScript
- 采集与 AI 服务：Python、FastAPI
- 主数据库：PostgreSQL
- 缓存与队列：Redis
- 搜索：首期 PostgreSQL 全文搜索，后续接入 OpenSearch
- 定时任务：Celery 或同类任务队列
- 对象存储：阿里云 OSS、腾讯云 COS 或兼容 S3 的存储
- 网页解析：HTTPX、Scrapy、正文抽取工具；动态页面按需使用 Playwright
- 语音识别：Whisper 或合规的云语音服务
- 部署：Docker、Nginx、云数据库和 CDN

### 7.2 推荐代码结构

```text
project/
├─ apps/
│  ├─ web/          # Next.js 用户端与管理端
│  ├─ api/          # NestJS 业务 API
│  └─ crawler/      # Python 采集与 AI 处理服务
├─ packages/
│  ├─ shared/       # 公共类型、常量和校验规则
│  └─ ui/           # 公共 UI 组件
├─ infra/           # Docker、Nginx 和部署文件
├─ docs/            # 接口、数据库和运维文档
└─ tests/           # 端到端和集成测试
```

## 8. 核心数据表

建议至少建立以下数据表：

- `users`：用户账号与资料
- `roles`、`permissions`、`user_roles`：角色权限
- `authors`、`organizations`：作者与机构
- `categories`、`tags`、`content_tags`：分类标签
- `articles`：新闻、政策和深度文章
- `article_sources`：文章来源与原始链接
- `videos`：视频元数据和内容概要
- `creators`：视频创作者与平台账号
- `posts`、`comments`：论坛帖子与评论
- `likes`、`favorites`、`follows`、`shares`：互动行为
- `events`、`event_registrations`：活动与报名
- `notifications`：站内通知
- `reports`、`moderation_logs`：举报和审核记录
- `crawl_sources`、`crawl_jobs`、`crawl_logs`：采集来源与任务
- `content_metrics`：阅读与平台互动指标
- `recommendation_scores`：热度与推荐结果
- `banners`、`site_settings`：运营配置
- `audit_logs`：后台操作日志

所有主要业务表建议包含 `created_at`、`updated_at`；软删除数据增加 `deleted_at`。

## 9. 安全与财经内容合规

- 全站使用 HTTPS。
- 密码使用安全哈希，验证码和登录接口限流。
- 管理员支持双因素验证。
- 数据库与 Redis 不直接暴露公网。
- 文件上传校验类型、大小和病毒风险。
- 富文本内容执行 XSS 清洗。
- 接口进行身份认证、权限检查和参数校验。
- 重要操作写入不可随意修改的审计日志。
- 每篇财经内容展示“仅供信息交流，不构成投资建议”。
- 禁止非法荐股、代客理财、收益承诺、虚假募资和内幕信息传播。
- 对“保本、带单、开户链接、老师荐股、加群”等内容触发拦截或人工审核。
- 认证作者和机构必须有审核记录，避免身份仿冒。
- 提供隐私政策、用户协议、社区规范、版权投诉和内容举报入口。

# 11. 一步一步搭建网站的 Prompt

## 11.1 使用方法

1. 按顺序一次只发送一个 Prompt 给编程助手。
2. 每一步完成后，先运行项目和测试，再进行下一步。
3. 如果已有代码，让编程助手先检查现有结构，不要重复创建或覆盖无关代码。
4. 密钥只放在本地环境变量或密钥管理服务中，不提交到 Git。
5. 每一步都要求同步更新必要的 README、环境变量示例和数据库迁移。

以下 Prompt 默认项目采用 Next.js + NestJS + Python FastAPI + PostgreSQL + Redis。

---

## 阶段 A：项目基础

### Prompt 1：检查开发环境

```text
请检查当前项目目录和可用的 Node.js、pnpm、Python、Docker 版本。只输出检查结果、缺失依赖和建议版本，暂时不要修改文件。
```

### Prompt 2：初始化项目结构

```text
请初始化一个 monorepo，包含 apps/web、apps/api、apps/crawler、packages/shared、packages/ui、infra 和 docs。加入基础 README、.gitignore 和环境变量示例，不实现业务功能。
```

### Prompt 3：初始化前端

```text
请在 apps/web 初始化 Next.js、TypeScript 和 Tailwind CSS。创建可运行的首页、全局布局、基础颜色变量和字体设置，品牌文案使用“洞察 · 链接 · 增长”。
```

### Prompt 4：初始化业务后端

```text
请在 apps/api 初始化 NestJS 项目，增加 /health 健康检查接口、统一响应格式、环境变量校验和全局异常处理。
```

### Prompt 5：初始化采集服务

```text
请在 apps/crawler 初始化 FastAPI 项目，增加 /health 接口、配置模块、结构化日志和基础测试，不开始抓取外部网站。
```

### Prompt 6：本地基础设施

```text
请创建本地 Docker Compose，启动 PostgreSQL 和 Redis。补充健康检查、持久化卷和环境变量示例，不写入真实密码。
```

### Prompt 7：前后端连通

```text
请让 Next.js 首页调用 NestJS 的 /health 接口，并显示后端服务状态。处理加载、成功和失败状态，补充本地启动说明。
```

---

## 阶段 B：数据库与账号系统

### Prompt 8：数据库基础配置

```text
请为 NestJS 接入 PostgreSQL 和数据库迁移工具。创建 users、roles、permissions、user_roles 基础表，并生成第一份迁移。
```

### Prompt 9：注册与登录 API

```text
请实现邮箱注册、邮箱密码登录、刷新令牌和退出登录 API。密码必须安全哈希，接口加入参数校验、限流和测试。
```

### Prompt 10：登录注册页面

```text
请实现前端登录、注册和退出功能，显示清晰的表单错误。登录状态需要在刷新页面后保持，并提供未登录访问保护。
```

### Prompt 11：角色权限

```text
请实现游客、用户、编辑、版主、运营管理员和超级管理员的角色权限控制。为受保护 API 增加权限守卫和最小测试。
```

### Prompt 12：用户中心

```text
请实现用户资料 API 和个人中心页面，支持修改头像、昵称、简介、行业、公司和职位。加入字段长度校验和隐私友好的默认值。
```

---

## 阶段 C：内容系统

## 12. 项目验收标准

网站达到以下条件后，可以视为第一期基本完成：

- 用户可以注册、登录和管理个人资料。
- 编辑可以发布新闻、政策、视频摘要和深度文章。
- 用户可以浏览、搜索、点赞、收藏、评论和关注。
- 用户可以在论坛发帖、回复和举报。
- 用户可以查看活动并报名。
- 系统可以从白名单来源发现、解析、去重和审核内容。
- 视频概要确实基于合法取得的字幕或转写内容生成。
- 资讯卡片包含标题、摘要、首图、来源和原文跳转。
- 推荐、最新、热榜和关注信息流可以正常使用。
- 管理后台可以审核内容、管理用户、配置首页和查看数据。
- 权限、限流、内容清洗、日志和备份可以正常工作。
- 网站可通过 HTTPS 在云服务器运行，并具备监控和恢复方案。

## 13. 后续需要确定的业务参数

正式开发前，项目方还需要最终确认：

- 网站正式名称、Logo 文件和品牌视觉规范
- 主要目标用户是投资人、企业家、从业者还是泛财经用户
- 首期重点行业及行业关键词词表
- 首批允许采集的网站清单
- 首批 Bilibili、YouTube、抖音 UP 主清单
- 文章来源和图片使用授权规则
- 国内云还是海外云，以及备案要求
- 是否首期接入微信登录、短信和微信服务号
- 是否允许完整转载，还是只展示摘要并跳转原文
- 是否需要会员、付费内容和机构认证
